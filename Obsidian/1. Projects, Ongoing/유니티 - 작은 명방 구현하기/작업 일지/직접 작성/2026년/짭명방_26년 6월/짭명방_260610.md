### 키 명명법 기준
- 내 프로젝트의 경우, 규모만 보면 사실 `Skill`을 구분할 필요도 없음. `EnemySkill`에는 보스 스킬 2개만 들어가니까.
- 근데 규모에 관계없이 이런 키를 명명하는 기준이 궁금해졌다. 
- 예를 들어서 보스가 폭발 스킬을 쓰는 경우랑, 아군 캐릭터가 폭발 스킬을 쓰는 경우를 테이블에 키를 설정하는 시점에서 구분할 필요가 있는가...에 대한 이야기.

>[!note]
>- 키 명명법 핵심 원칙 : **키는 "누가 쓰느냐"가 아니라, "무엇인가"를 기술해야 한다.** 
>	- 해당 컨텐츠의 정체성을 나타내야 한다.
>	- **사용 주체나 맥락은 원칙적으로 포함하지 않는다.** 나중에 동일한 스킬을 다른 주체가 쓰거나 다른 상황에서 쓰게 된다면 키가 거짓말을 하는 것이 되기 때문.

- 기본적인 구조는 아래처럼 가져가면 됨.
```
{DOMAIN}_{ID}_{TYPE}

ex)
{SKILL}_{INFRENO_BLAST}_{NAME}
```

#### 참고
- SO 파일의 명명법은 아래처럼 가져감
```
{Type}_{ID}

ex)
SkillData_FireBall
```
> 유니티 인스펙터 / 프로젝트 창에서 타입 prefix로 필터링이 쉬워짐

### 따라서 Skills 테이블 하나에 모든 스킬을 집어넣음
- 명명 규칙 : 이전에 `Operator`에서 한 것과 비슷하게 스크립트로 처리해주면 된다. 

- 기존 구조가 이랬다면
```cs
[SerializeField] protected string skillName = string.Empty;
public string SkillName => skillName;
```

- `skillName`을 `id`로 변경하고, `nameKey`랑 `descKey`는 규칙으로 얻게 하면 된다.
```cs
public string id;

public string NameKey => $"SKILL_{id}_NAME";
public string DescKey => $"SKILL_{id}_DESC";

public string GetName() => LocalizationSettings.StringDatabase.GetLocalizedString(LocalizationManager.Skills.TableName, NameKey);

// 불러오는 구조는 같은데 스킬마다 달라질 수 있음
public virtual string GetDesc() 
{
}
```
> `id, Name`은 공통 구현이지만 `Desc` 관련 부분은 `OperatorSkill`에만 있는 부분이다. 여기에는 편의상 한꺼번에 적어둠

- 관련 패널들 수정 : 원래 `SkillName`이나 `Description`을 참조하던 부분들을 `GetName()`, `GetDesc()`으로 가져오게 변경

- 스킬들의 `Description`을 `String Table`로 옮기는 작업 시작. 이 부분이 오래 걸릴 거다..
