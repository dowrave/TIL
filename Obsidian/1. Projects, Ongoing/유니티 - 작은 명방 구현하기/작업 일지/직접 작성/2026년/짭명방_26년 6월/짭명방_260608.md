- 구현 방향
	- `UpdateStatContainerTitle`의 경우, 하단의 `valueTypeTitle`이랑 구분을 해서 넣어야함
	- 그래서 `statTypeTitle`을 스마트 스트링으로 구현한 다음에, `valueTypeTitle`값을 가져와서 `argument`로 전달하는 방식이 될 것 같음

- "수치"라는 용어를 "총량"으로 변경함. 
	- `Numeric`이라는 단어를 의식하고 만든 건데, 그것보다는 `Value`나 `Total`이 더 자연스럽다는 듯. 이외에도 버프 등에서는 고정 수치를 표현하는 `Flat` 같은 표현도 쓴다고 한다.

- **생각 : 로컬라이제이션 키값**
	- 캐릭터의 스탯도 `STAT`이라는 키워드를 쓰고, 통계 패널도 `STAT`이라는 키워드를 써서 겹친다. 둘 다 `UI` 테이블에서 관리되고 있기도 함.
	- 키값 명명 규칙에 대해
		- `CHARACATER_NAME_DETAIL`이 맞을까? `CHARACTER_DETAIL_NAME`이 맞을까?
		- **`CHARACTER_DETAIL_NAME`이 맞다고 한다. 기존 구조 유지함.**
			- 키 이름은 계층 구조로 읽혀야 한다. `CHARACTER_NAME_DETAIL`이 되면 "캐릭터의 이름의 디테일"이라는 구조가 됨.
				- `CHARACTER_DETAIL_NAME`
				- `CHARACTER_DETAIL_DESCIPRTION`
				- `CHARACTER_DETAIL_STAT_ATK`
				- `CHARACATER_DETAIL_STAT_DEF`
			- 정렬 관점에서도 `CHARACTER_DETAIL_*` 처럼 검색하기 편함.
				- 안 해봤다. AI가 그렇다길래 일단 달아만 놓음.
	- `STAT`이 겹치는 문제는, 캐릭터의 속성은 `STAT`으로, 통계 개념은 `STATS`으로 가져가기로 함.
		- 테이블을 새로 만들고 관련 항목들을 정리하는 것도 생각해봤는데, 이미 스크립트와 연결된 구조라는 점 + 테이블을 옮기는 작업이 노가다라는 점 등등 때문에 **규칙만 만든다는 개념**으로 작업하겠음.
		- 기존의 **유닛 스탯 개념** : `UI_STAT` => **`UI_UNIT_STAT`**
		- **통계 개념 : `UI_STATS`**

- 생각 2 : 스마트 스트링을 써야 할까? 안 써도 될까?
	- 표시되어야 하는 텍스트는 `A(B)`라는 구조이고, 두 알파벳 모두 다른 키값에서 가져오는 개념.
	- **지금 구현하는 언어들에서는 저 구조가 크게 달라지진 않기 때문에 저렇게 구현해도 무방**함.
	- 근데 언어에 따라서 순서가 달라진다거나, 변주를 주고 싶을 때도 있을 거임. 이런 경우까지 고려한다면 `적에게 가한 피해량({metric})` 같이, A에 해당하는 키가 전체적인 구조를 결정하고, B에 해당하는 키가 그 안에 들어가는 식으로 스마트 스트링으로 구현할 수도 있음.
	- 굳이 할 필요는 없어보이지만... 그래도 이렇게 구현해봄.

### `EnemySpawner`에서 `null`로 출력되던 부분
```cs
if (spawnedObject == null)
{
	Logger.Log($"[EnemySpawner] {tag}에 해당하는 SpawnedObject = null");
	return;
}
```
> - 이 부분은 게임이 끝났는데도 `Spawner`가 계속 동작하기 때문에 발생하는 문제
> - 게임이 끝난 다음에 오브젝트 풀에 있던 오브젝트가 파괴 -> 이후에 호출하려고 해서 발생
> - 게임 종료 후에는 이 `Spawn()`을 호출하는 `SpawnEntites`의 동작을 멈추면 될 듯

```cs
private IEnumerator SpawnEntities()
{
	if (spawnList.Count == 0)
	{
		yield break;
	}
	float startTime = Time.time;

	foreach (var spawnData in spawnList)
	{
		yield return new WaitForSeconds(spawnData.SpawnTime - (Time.time - startTime));

		// 스폰하기 전에 게임이 끝났는지 여부를 체크, 끝났다면 동작 중지
		if (StageManager.Instance.CurrentGameState == GameState.GameWin || 
			StageManager.Instance.CurrentGameState == GameState.GameOver)
			yield break;

		Spawn(spawnData);
	}
}
```

### `StageScene`에서 `OptionPopup` 켰을 때
```
NullReferenceException: Object reference not set to an instance of an object
KeyBindingSection.ResetBindingSettings () (at Assets/Scripts/UI/Popups/OptionPopupComponents/KeyBindingSection.cs:142)
KeyBindingSection.OnDisable () (at Assets/Scripts/UI/Popups/OptionPopupComponents/KeyBindingSection.cs:176)
UnityEngine.GameObject:SetActive(Boolean)
OptionPopup:Start() (at Assets/Scripts/UI/Popups/OptionPopup.cs:37)
```

저 37번 줄이 아래 코드.
```cs
	tabEntry[i].section.SetActive(false);
```

1. 각 `section`은 비활성화해야 하는 게 맞음 : 모든 패널이 보이면 안 되니까.
2. 저 에러가 왜 발생했냐면 참조 변수로 저장하는 `keyBindingManager`의 초기화가 제대로 이뤄지지 않았기 때문임
```cs
private void Start()
{
	keyBindingManager = GameManagement.Instance.KeyBinding;
}
```

- `OptionPopup.OnDisable()`이 `KeyBindingSection.Start()`보다 먼저 동작하고 있는 상황.
- `KeyBindingSection`에서도 `Start()`에서 초기화가 이뤄진다.

그래서 어떡하나 고민이 됐는데, 생각보다 심플하다. `keyBindingManager`의 참조 조건만 체크하면 됨.

```cs
private void ResetBindingSettings()
{
	ResetWaitingContainer();
	ResetDuplicatedContainer();
	if (keyBindingManager != null &&  // 이 조건 추가
		keyBindingManager.IsRebinding)
	{
		StartCoroutine(StopRebindingNextFrame());
	}
}
```
> - `StopRebindingNextFrame()`은 키 바인딩 상태를 한 프레임 이후에 해제하는 코드임
> - 지금의 문제 상황은 옵션 팝업을 조작하다가 해제되는 게 아니라 최초 설정 중에 섹션을 비활성화하면서 발생하고 있다. 즉, **별도의 바인딩 동작이 없는 상태 = 초기화이므로 null 체크만으로 처리하는 것도 가능**함
