## EnemyStunState 구현
- `State`로 만들어야 하는 CC : 행동 자체에 제약이 걸리는 경우
	- `Stun, Freeze, Sleep` 등등..

- 어떻게 작업함?
1. 상태 머신에 강제 상태 전환 메서드 추가
```cs
public void ForceChangeState<T>() where T : EnemyState
{
	if (_currentState == _states[typeof(EnemyDeathState)]) 
	{
		// 이미 죽었으면 상태 전환 불가
		return; 
	}

	Logger.Log($"[강제 상태 전환(Interrupt)] 기존 : {_currentState} / 전환 : {_states[typeof(T)]}");
	
	_currentState?.Exit();
	_currentState = _states[typeof(T)];
	_currentState.Enter();
}
```

2. 스턴 진입 : `BuffController`의 이벤트를 받아 FSM 단위에서 강제로 덮어씌움

- 아래의 기타 이슈 수정에서 메서드 수정하고 여기로 옴
```cs
protected override void HandleBuffAdded(Buff buff)
{
	base.HandleBuffAdded(buff);
	if (buff is StunDebuff)
	{
		stateMachine.ForceChangeState<EnemyStunState>();
		
	}
}
```

3. 디버프 VFX를 띄우는 위치
	- 기존에는 `StunDebuff`에서 `DebuffVFXMAnager`를 호출하는 방식이었음
	- `StunState`에서 구현할지, 아니면 지금처럼 `Buff` 단위에서 처리해버릴지가 고민이다.
	- 위에서 구현한 `HandleBuffAdded`에 VFX 실행 코드 추가.
- 즉 버프 컨트롤러에서 이벤트를 발생시키면, 컨트롤러를 가진 객체인 `UnitEntity`에서 이를 구독, `HandleBuffAdded 및 HandleBuffRemoved`이 실행된다.
	- `UnitEntity` 자체에서는 이를 받아서 `DebuffVFXManager`의 `VFX`를 실행시킴
	- `UnitEntity`의 자식 클래스 `Enemy`에서는 강제 상태 변화를 실행시킨다.
	- `StunState` 자체는 크게 구현할 게 없음.
```cs
public class EnemyStunState: EnemyState
{
    public EnemyStunState(EnemyStateMachine stateMachine)
        : base(stateMachine) // 부모 클래스의 생성자를 먼저 호출한 다음 본문의 생성자를 호출하는 문법
    {
        
    }

    public override void Enter()
    {
        base.Enter();

        // [구현할 필요 없음]
        // 기존 로직 취소? - State가 달라지므로 기존의 Move나 Attack, Cast는 실행되지 않음
        // -- 만약 이슈가 있다면 다른 State의 Exit 부분을 체크해봐야 할 듯.
        // 기절 이펙트 재생 : Enemy에서 스턴 버프가 생성될 때 바로 실행되도록 구현되어 있음
    }

    public override void Update()
    {
        base.Update();

        // 1. 스턴 상태 유지 조건 체크
        bool isStillStunned = _owner.Buff.HasBuff<StunDebuff>();

        // 2. 스턴이 끝났다면 다음 상태로 전환
        if (!isStillStunned)
        {
            DetermineNextState();
        }
    }

    public override void Exit()
    {
        base.Exit();
    }

    // 스턴이 끝난 후 이동할 상태 결정
    private void DetermineNextState()
    {
        if (EnemyStateTransitionCondition.CanUseSkill(_owner))
        {
            _fsm.ChangeState<EnemySkillCastState>(this);
        }
        else if (EnemyStateTransitionCondition.CanAttack(_owner)) 
        {
            _fsm.ChangeState<EnemyAttackState>(this);
        }
        else
        {
            _fsm.ChangeState<EnemyMoveState>(this);
        }
    }
}
```

## EnemyDeathState 구현
- 체력이 0이 됐을 때 바로 사망 처리하지 않고, "**객체에서 FSM을 Death로 전환 -> Death의 Enter에서 사망 로직이 실행 -> 객체의 사망 로직 실행**"되도록 상태로 권한을 위임해야 한다.
> 크게 내가 고민할 부분은 아닌 것 같아서 제미나이에게 맡겼음.
```cs
// protected override void HandleOnDeath() => Despawn(EnemyDespawnReason.Defeated);
protected override void HandleOnDeath()
{
	// 바로 Despawn을 실행하지 않고, 상태 머신을 Death로 전환함(다른 상태의 동작을 한 번 막음)
	stateMachine.ForceChangeState<EnemyDeathState>();
}

// 사라지는 로직 처리
// protected void Despawn(EnemyDespawnReason reason)
public void ExecuteDespawnProcess(EnemyDespawnReason reason)
{        
	// 예외 처리
	if (currentEnemyDespawnReason != EnemyDespawnReason.Null) return;

	currentEnemyDespawnReason = reason;

	// UI 제거
	if (enemyBarUI != null)
	{
		enemyBarUI.gameObject.SetActive(false);
	}

	// 디스폰 이유 파라미터를 포함하는 이벤트 발생
	OnEnemyDespawned?.Invoke(this, currentEnemyDespawnReason);

	// 사망 처리
	DieWithAnimation(); 
}
```
> 기존에는 HandleOnDeath이 곧바로 Despawn을 실행시켰음.
> 지금은 HandleOnDeath는 상태 전환만, ExecuteDespawnProcess는 DeathState에서 호출하는 방식으로 바뀌었다. 
> 추가로 하나 생긴 효과라면 **DeathState가 되면서 Move나 Attack을 하면서 사라지지는 않게 됐다는 것.**

## ActionRestriction 관련
- 기존에 구현된 형태임.
```cs
// 행동에 제약이 걸리는 조건들을 정리합니다.
[System.Flags] // enum이 비트 필드 = 여러 상태의 조합으로 사용될수 있음을 나타냄
public enum ActionRestriction
{
    None = 0,
    Stunned = 1 << 0, // 00000001 (1)

    // 아래는 참고로 쓰라는 예제들
    // Frozen = 1 << 1,   // 빙결 상태 (2)
    // Disarmed = 1 << 2, // 무장 해제 (공격만 불가) (4)
    // Rooted = 1 << 3,   // 속박 (이동만 불가) (8)
    // Casting = 1 << 4,  // 스킬 시전 중 (16)

    CannotAttack = Stunned,
    CannotMove = Stunned, 
    CannotUseSkill = Stunned, 
    CannotAction = Stunned, 
}
```

- "더 이상의 CC를 추가하지 않는다"를 고려해도, 이게 지금은 굳이 필요한 상태는 아니게 됐음. 왜냐하면 각 State가 이미 그 역할을 해주고 있기 때문임.
