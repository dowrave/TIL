## Enemy FSM 계속

### Move에서 Attack으로의 전환
- 크게 2가지 상황이 있음
1. 근거리에서 저지당한 경우 
2. 원거리인데 공격범위 내에 적이 있는 경우

- 1번의 경우, 이동은 어차피 불가능한 상황이므로 가만히 있음
- 2번의 경우, 공격 쿨타임 동안은 이동할 수 있음

- 어느 정도 수정한 다음에 글로 남긴다.
- 일단 공격 컨트롤러를 바탕으로 생각해보면
	1. 공격 컨트롤러에서 처리했던 기존 상황들에 반환값을 추가
	2. 이동이 가능한 경우는 공격 상태에서 이동 상태로 전환
```cs
public AttackResult OnUpdate()
{
	// 공격 범위 내의 적 리스트 & 현재 공격 대상 갱신
	SetCurrentTarget();

	if (AttackDuration > 0f) return AttackResult.StayInAttack;

	// 저지당함 - 근거리 공격
	if (_blockingOperator != null)
	{
		HandleAttackBlockingOperator();
		return AttackResult.StayInAttack;
	}

	// 저지당하지 않음
	else
	{
		// 바리케이드를 공격하는 상황(바리케이드는 공격 대상에 포함되지 않음)
		if (HandleAttackBarricade()) return AttackResult.StayInAttack;

		// 공격 대상이 있음
		if (HandleAttackTarget()) return AttackResult.StayInAttack;
	}
	
	// 이동할 수 없는 상황을 모두 처리했음
	// 나머지 상황은 이동 가능
	return AttackResult.CanMove;
}
```

```cs
public class EnemyAttackState: EnemyState
{
    private EnemyAttackController _enemyAttackController;
    
    public EnemyAttackState(EnemyStateMachine stateMachine)
        : base(stateMachine) // 부모 클래스의 생성자를 먼저 호출한 다음 본문의 생성자를 호출하는 문법
    {
        _enemyAttackController = _owner.Attack;
    }

    public override void Update()
    {
        AttackResult result = _enemyAttackController.OnUpdate();

        // 이동할 수 있는 상황이라면 이동 상태로 전환
        if (result == AttackResult.CanMove)
        {
            _fsm.ChangeState<EnemyMoveState>();
        }
    }
}
```

- 이외에도, 공격 가능한 상태인지를 체크, 대상이 있다면 공격 상태로 전환하는 로직도 필요함.
- 이 로직을 수행하는 쪽은 `상태 머신`이 된다. 
	- **모든 상태에서 동일하게 적용되는 규칙**이기 때문이다. 
		- 즉 `Any State -> Attack`에 대한 구현은 머신에서 하면 됨.
	- "감지해서 공격 대상에 추가한다"는 이미 센서 단위에서 진행되고 있는 일이므로, 이후 동작들만 생각하면 됨.

- `FSM`에서 구현을 이렇게 가져갈 수 있음.
```cs
    public void Update()
    {
        // 공격 상태로 전환되어야 하는 상황일 때는 공격 상태로 전환
        if (_currentState is not EnemyAttackState && CanTransitionToAttackState())
        {
            ChangeState<EnemyAttackState>();
        }

        _currentState.Update();
    }

    private bool CanTransitionToAttackState()
    {
        return _owner.Attack.TargetsInRange.Count > 0 
        || _owner.Attack.IsInAttackBarricadeDistance(); // 바리케이드가 null이 아니면서 0.5f 이내일 대
    }

```
- 바리케이드의 조건 처리 관련
	- "대상 바리케이드가 있다"는 것만으로 공격 상태로 전환할 이유가 되진 않고, "대상 바리케이드가 있고, 현재 위치에서 바리케이드까지의 위치" 조건이면 될 듯.
	- 뭔가 구질구질하게 길어지는 느낌이지만 일단은 이렇게 가져가기로.

- 테스트 후에 보스 관련 처리도 작업 진행함

### 테스트
1. 저지당하는 상황에서도 이동이 멈추지 않음
	- `_blockingOperator` 필드와 메서드들의 위치를 `Enemy`로 이동(여러 상태가 같이 쓰기 때문)
	- `BlockingOperator` 여부에 따른 처리 추가
```cs
    public MoveResult OnUpdate()
    {
		// ...
		
		// 저지당했을 때의 동작
        if (_owner is Enemy enemy && enemy.BlockingOperator != null)
        {
            return MoveResult.Blocked;
        }
	}
```

2. 원거리 적은 공격을 시작하면 쿨다운 동안에는 이동해야 하는데 아예 이동하지 않음
	- 상태 추적 시 `CanMove`로 결과값이 나오고 상태 전환도 잘 진행되는 것으로 보임
	- 그럼 곧바로 다시 `AttackState`로 돌아간다는 의미다.
```cs
// 공격 상태로 전환되어야 하는 조건들
private bool CanTransitionToAttackState()
{
	// 저지당했을 때
	if (_owner.BlockingOperator != null) return true;

	// 공격 대상이 있고, 현재 공격이 가능한 상태일 때
	if (_owner.Attack.TargetsInRange.Count > 0 && _owner.Attack.AttackCooldown <= 0f) return true;

	// 타겟 바리케이드가 있고, 공격할 수 있는 위치에 있을 때
	if (_owner.Attack.IsInAttackBarricadeDistance()) return true;

	return false;
}
```
> 공격 쿨다운 체크도 상태 머신에서 진행하도록 추가함

### Skill 사용 처리
- `EnemyBoss`의 경우, 일반적인 `Enemy`의 패턴을 따르는 것 외에도 추가로 스킬을 사용할 수 있음
- 공격 / 스킬을 모두 쓸 수 있는 상황이라면 스킬 사용을 우선함

- **`AttackState`에서 통합** : `Enemy`인지 `EnemyBoss`인지에 따라 다형성으로 처리하기만 하면 됨
- 이름은 `CombatState`로 바꿈

- 왜 통합해도 되는가?
	- **현재 `State`에서 하는 역할은 `OnUpdate`를 실행하고, 상태를 `Move`로 바꾸냐 마냐**임. 즉 `OnUpdate`에서 어떤 걸 실행하는가는 전혀 중요하지 않음.

- `EnemyCombatState.cs`
```cs
public class EnemyCombatState: EnemyState
{
    private EnemyAttackController _enemyAttackController;
    private EnemySkillController _enemySkillController;
    
    public EnemyCombatState(EnemyStateMachine stateMachine)
        : base(stateMachine) // 부모 클래스의 생성자를 먼저 호출한 다음 본문의 생성자를 호출하는 문법
    {
        _enemyAttackController = _owner.Attack;
        if (_owner is EnemyBoss _boss)
        {
            _enemySkillController = _boss.SkillController;
        }
    }

    public override void Update()
    {
        // Boss인 경우는 스킬 처리
        // if (_enemySkillController != null)
        // {
        //     CombatResult skillResult = _enemySkillController.OnUpdate();
        //     if (skillResult == CombatResult.CanMove)
        //     {
        //         _fsm.ChangeState<EnemyMoveState>();
        //         return;
        //     }
        // }
        
        // 스킬 시도(보스가 아니라면 스킬 판단은 항상 "할 일 없음")
        CombatResult skillResult = _enemySkillController?.OnUpdate() ?? CombatResult.CanMove;

        if (skillResult != CombatResult.CanMove)
        {
            // 스킬이 실행 중이거나 쿨다운 대기 상태일 때 상태를 유지, Attack은 확인할 필요 없음
            return;
        }

        // 스킬에서 할 일이 없었다면 Attack을 확인
        CombatResult attackResult = _enemyAttackController.OnUpdate();
        if (attackResult == CombatResult.CanMove)
        {
            _fsm.ChangeState<EnemyMoveState>();
        }
    }
}
```

// 사용할 수 있는 스킬이 있는지를 점검하는 조건 점검 메서드
public bool HasUsableSkill()
{
	if (_currentGlobalCooldown > 0f) return false;

	List<EnemyBossSkill> skillsToCheck = (_owner.BlockingOperator != null) ? meleeSkills : rangedSkills;

	foreach (EnemyBossSkill skill in skillsToCheck)
	{
		if (skillCooldowns[skill] <= 0 && skill.CanActivate(_owner))
			return true;  
	}
	
	return false;
}
```

// 전투 상태로 전환되어야 하는 조건들
private bool CanTransitionToCombatState()
{
	// 저지당했을 때
	if (_owner.BlockingOperator != null) return true;

	// 스킬 사용 가능할 때

	if (_owner.Attack.IsInAttackBarricadeDistance()) return true;

	return false;
}
```


## AI 대화 요약본

> 얘한테 알리지 않고 내가 수정한 내용도 있긴 한데, 대부분은 들어가 있을 듯?
> Claude Sonnet 5을 사용했다.

### 1. FSM에서 "감지"와 "전환 판단"의 책임 분리

- **감지(Sensor)**: 상태와 무관하게 항상 동작, 사실만 보고 (`OnTriggerEnter/Exit` → `_targetsInRange` 갱신 등)
- **전환 판단**: 조건의 스코프에 따라 위치가 달라짐
    - 여러 상태에 공통되는 전역 조건(예: "범위 내 타겟 감지되면 전투 상태로") → **FSM 레벨**에서 처리 (Animator의 "Any State" 트랜지션과 동일 개념)
    - 특정 상태에서만 의미 있는 지역 조건(예: "공격 애니메이션 끝남") → **해당 State 내부**에서 처리
- Sensor가 직접 `ChangeState()`를 호출하면 안 되는 이유: 감지 컴포넌트가 전환 정책까지 떠안게 되어 책임이 섞임
> `State`에서 객체나 센서 등의 상태를 보고 자신의 상태를 전환하게 구현하는 쪽이 맞음


### 2. `EnemyAttackController.OnUpdate()` 코드 리뷰

- `CurrentTarget`은 공격의 **필요조건이지 충분조건은 아님** (쿨다운/모션/`_stopAttacking`도 함께 봐야 함)
- 바리케이드 공격 경로는 `CurrentTarget`과 무관한 **별도 분기**라는 점을 확인
- `HandleAttackBlockingOperator` / `HandleAttackBarricade` / `HandleAttackTarget`으로 함수 분리
- 리팩터링 과정에서 바리케이드 분기에 `CanAttack()`을 넣으면 `CurrentTarget == null`일 때 바리케이드 공격이 막혀버리는 **회귀 버그 가능성**을 짚음
- `AttackDuration > 0f`을 최상단 게이트로 뺀 것은 타당함 (후딜 중엔 무조건 대기)

### 3. 반환 타입: `bool` vs `enum`

- 판단 기준: "호출부가 반환값에 따라 서로 다른 후속 동작을 해야 하는가"
- 지금 목적(Move 전환 여부)은 사실상 이진값이라 2값 `enum`(`CanMove` / `StayInCombat`류)이면 충분, 4값까지는 불필요 (YAGNI)
- `None` 기본값 포함 여부: 함수가 항상 명시적으로 반환하고 그 자리에서 바로 소비되는 구조라면 불필요, 필드 저장/직렬화/Inspector 노출되는 경우라면 방어용으로 필요

### 4. `EnemySkillController` 추가 및 `EnemyBoss` 확장

- `EnemyBoss`만 갖는 스킬 로직을 위해 **별도 State를 만들지 않고**, `Enemy`/`EnemyBoss`의 **다형성**으로 위임하는 방향 결정
- State(`AttackState` → `CombatState`로 개명)는 "전투 행동을 위임하고 이동을 멈춘다"는 역할만, 스킬이냐 공격이냐는 몰라도 됨
- FSM도 Enemy 타입을 분기하지 않고 `Enemy.CanTransitionToCombat()` 같은 가상 메서드에 위임

### 5. Combat ↔ Move 오실레이션 버그 두 가지 발견 및 수정 방향

- **원인 1**: 전환 조건(`CanTransitionToCombatState`)이 전체 쿨다운만 확인하고, 실제 실행 로직(`TryUseSkill`)은 개별 스킬 쿨다운까지 확인 → 기준 불일치로 Combat 진입 후 곧바로 되돌아 나가는 반복 발생
    - 해결: `HasUsableSkill()` 같은 단일 판단 메서드를 만들어 전환 조건과 실행 조건이 **동일한 소스**를 참조하도록 통합
- **원인 2**: `EnemyCombatState.Update()`에서 스킬이 `CanMove`를 반환하면 Attack을 확인하지도 않고 즉시 Move로 전환 → "스킬 우선, 안 되면 공격" 설계와 어긋남
    - 해결: 스킬이 `CanMove`일 때만 이어서 Attack도 체크하도록 흐름 수정
