### 배치 동작 중에 오퍼레이터가 제거되었을 때 배치 동작에 버그 발생
- 일단 원하는 구현, 필요한 것들을 정의해놓고 시작해야 할 듯

>[!note]
>- 원하는 구현 : **현재 진행되고 있는 배치 동작이 방해받으면 안됨**
>- 필요한 것 : **배치된 요소가 제거된 자리에도 배치가 활성화**되어야 함
>- 현재 상황
>	- 오퍼레이터 드래그 중 제거되었다면 배치 상태가 캔슬됨
>	- 이 상태에서 드래그 중이었던 커서를 떼면 `DeployableBox`가 제거되고 다시 나타나지 않음

- 수정 방향
1. **배치 상태를 캔슬하는 로직을 제거**
2. 현재 배치 가능한 타일을 업데이트
```cs

// 현재 배치 중인지에 관한 변수로 정리
public bool IsDeploying => CurrentDeployableBox != null;

public void HandleDeployableRemoved(DeployableUnitEntity deployable)
{
	deployedItems.Remove(deployable);

	// 1번에 해당 - 주석 처리
	// 현재 deployable에 대한 정보를 UI에서 보여주고 있다면 숨김
	// StageUIManager.Instance.HideInfoPanelIfDisplaying(deployable);
	// HideOperatorUIsOnCondition(deployable);
	// ResetHighlights();

	// 2번에 해당
	// 배치 중일 때의 처리(현재 박스가 있다 = 화면 하단의 박스 UI를 클릭했다)
	if (IsDeploying)
	{
		HighlightDeployableTiles();
	}
```
> - 현재 **배치되지 않은 요소를 배치 중인지에 대한 상태 변수를 하나 추가**했다. `CurrentDeployableBox != null`이 "현재 배치 중인가"를 의미한다는 걸 지금은 알지만 나중에 봤을 때 왜 그런지 기억이 나지 않을 수 있기 때문.
> - 잘 반영된 듯

#### 변수명 처리의 우선순위
- `CurrentDeployableBox != null`같이 현재는 이해할 수 있지만 미래에 다시 봤을 때, 혹은 타인에게 보여줬을 때 그 **의미를 곧바로 이해하기 어려운 것들을 `IsDeploying` 같이 의미를 이해할 수 있는 변수로 정리해두는 게 프로젝트의 우선순위에서 어느 정도에 위치하는가?**

- **습관 자체를 "처음부터 좋은 이름을 짓기"로 정하는 게 좋음**

1. 기능의 구현보다는 기능의 완성 과정에 가까움. **구현보다는 우선순위가 떨어지는 건 맞음.**
	- 그래서 프로토타이핑 단계에서의 코드라면 굳이 이런 과정을 거칠 필요는 없음
2. **하지만 분명 중요한 요소임**
	- 지금 코드를 잘 써 두면, 나중에 읽을 때마다 헷갈리는 시간을 아낄 수 있음
	- `CurrentDeployableBox != null`의 의미를 기억하는 시간이 길지 않음. 몇 주 후든, 다른 기능을 구현하고 돌아온 뒤든 사람의 뇌에서 그 맥락이 휘발됨
	- 게다가 프로퍼티이기 떄문에 다른 클래스에도 쓸 가능성이 있는 코드이므로, 이런 식으로 정리해두는 게 유용함.


### 로비로 돌아가기 패널이 꺼졌는데 게임이 재생이 안됨
- `ReturnToMainMenuPanel`의 초기화 이슈.
```cs
    public void Initialize()
    {
		// ...
        previousState = StageManager.Instance.CurrentGameState; // 이게 없었음..
```

- ...가 아니었다. `StageUIManager`에서 `Pause`로 바꾸고 패널을 초기화하기 때문에 발생한 문제. 이전 상태를 `Pause`로 기억할 수밖에 없다.

1. 매니저에선 단순히 `gameObject.SetActive(true)`로 바꾸고
2. **패널에선 `Initialize()`와 `OnEnable`로 나뉘었던 요소들을 `Initialize`로 통합함.**
```cs
private void OnEnable()
{
	if (StageManager.Instance != null)
	{
		previousState = StageManager.Instance.CurrentGameState;
		StageManager.Instance!.SetGameState(GameState.Paused);  
	}

	AddOnClickEventListeners();

	// 단축키 사용 비활성화
	GameManagement.Instance.InputCommand.SetEnabled(false); 

	// 애니메이션
	if (canvasGroup != null)
	{
		canvasGroup.alpha = 0f;
		canvasGroup.DOKill();
		canvasGroup.DOFade(1f, animationSpeed)
			.SetUpdate(true); // Time.timeScale 무시
	}
}
```
> 지금 보니까 파라미터를 전달할 필요가 있는 게 아니라면 `OnEnable()`로 다 뭉쳐도 될 듯 싶다. 그래도 전체적인 리팩토링까진 안 할 거임.

3. `OnDisable`에도 `previousState`로 되돌리는 코드 추가. 끝.

> - `affectTargets`을 순회하면서 `Remove`를 하므로 열거 도중에 수정되는 현상
> - 제거할 대상을 별도의 리스트에 모아둔 뒤, 순회 후에 한꺼번에 제거하는 방법이 추천됐다. 
> 	- `.ToList()`로 스냅샷을 뜨는 방법도 있다. 매 프레임 호출되는 경우는 위 방법을 더 권장함.

- 여기서는 별도의 리스트에 모은 뒤 한꺼번에 제거하는 방법으로 처리
```cs
protected override void CheckTargetsInField()
{
    // 이전에 힐을 받던 대상들 중 범위를 벗어난 경우 처리
    List<UnitEntity> targetsToRemove = null;

    foreach (var target in affectedTargets.Keys)
    {
        if (!IsTargetInField(target))
        {
            foreach (var effect in affectedTargets[target])
            {
                target.RemoveBuff(effect);
            }

            targetsToRemove ??= new List<UnitEntity>();
            targetsToRemove.Add(target);
        }
    }

    if (targetsToRemove != null)
    {
        foreach (var target in targetsToRemove)
        {
            affectedTargets.Remove(target);
        }
    }
```
