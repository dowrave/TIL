## Enemy FSM 구현
- **[[유한 상태 머신(FSM)]] 및 [[짭명방_260710]] 참조**
### 상태로 만들 것과 조건/변수로 만들 것
#### 상태로 만들어야 하는 요소 체크 리스트
- [ ] **독립적인 행동 루프**가 있는가?
	- **`Update()` 안**에서 그 자체로 완결된, **다른 상태와 구분되는 로직**(애니메이션, 이동, 판정)이 있는가?
	- 예시 : "`Attack`은 타겟을 향해 대미지를 주기적으로 가한다", "`Move`는 다음 노드를 향해 이동한다"

- [ ] **다른 상태와 동시에 존재할 수 없는가?**
	- 여러 상태가 동시에 켜질 수 있다면 그건 상태가 아니라 상태 위에 별도로 놓이는 축이다.
	- 예시 : `isBlocked`는 이동 중에도, 공격 중일 때도 계속 참일 수 있는 값 -> 상태가 아니라 조건/변수로 처리하는 게 맞음

- [ ] **진입/퇴장 시점에 고유한 처리`Enter/Exit`가 필요한가?**
	- 애니메이션 트리거, 사운드, 파티클, 특정 타이머 초기화 등, 상태에 들어오거나 나갈 때에만 하는 처리가 있으면 상태로 구현하는 걸 고려할 법함
	- 단순히 변수를 켜고 끄는 정도라면 조건으로 구현

- [ ] 이 **개념이 늘어나면 상태 조합이 기하급수적으로 늘어나는가?**`(상태 폭발 테스트`)
	- `Attack_Blocked`, `Move_Slowed`, `Attack_Slowed` 같이 조합이 늘어난다면 설계가 잘못됐다는 의미

- [ ] 외부(타 시스템, 애니메이터, UI)에서 지금 어떤 상태인지를 구분해서 알아야 하는가?
	- 애니메이터 컨트롤러가 `Idle/Move/Attack/Death` 등만 구분하면 된다면 실제 상태 갯수의  좋은 힌트다.

#### 조건/변수로 처리할 요소
- 상호 배타적이지 않고, 여러 상태에 걸쳐 지속적인 값
- 상태 진입 시 무엇을 할지를 결정하는 파라미터
- 여러 상태의 전이 조건 계산에 공통으로 쓰이는 값

#### 애매한 경우의 판단 기준
- `Stun` : 아무 것도 못하고 멈춰 있는 상태 -> **기존의 모든 상태 로직을 무시하고 아무것도 하지 않음이라는 독립적인 행동이 있기 때문에, 별도의 상태**로 만드는 게 타당함
- `Slow` : 이동/공격 속도만 낮아질 뿐 **`Move/Attack` 등의 로직은 그대로 이므로 조건**으로 처리

> 저지당했을 때는 그냥 `공격` 상태로 변경하면 됨. 

### State와 Controller의 역할 나누기
- **`Controller`에서는 기능이 어떻게 동작하는지**만을 저장함
- **`State`에서는 `Controller`의 동작 및 그 결과를 받은 이후의 처리를 결정**함
	- "`Controller`의 동작 결과"에 해당하는 타입도 하나 정의해두면 좋음. 
	- 즉, 컨트롤러 자체에서 상태의 전이를 일으키지 말라는 의미임.

- 일단 오늘은 여기까지. 
```cs
public MoveResult OnUpdate()
{
	// 의도적인 대기 중일 때는 움직이지 않음
	if (_isWaiting) return MoveResult.Waiting; 

	// 이동 경로 중 도달해야 할 곳이 없다면 false 반환 (갈 곳이 없으니 이동이 불가능)
	if (_owner.CurrentPathIndex >= _owner.CurrentPathPositions.Count) return MoveResult.NoPath;

	Move(_owner.CurrentDestination);
	_owner.RotateModel();

	// 노드 도달 확인
	if (Vector3.Distance(_owner.transform.position, _owner.CurrentDestination) < 0.05f)
	{
		// 목적지 도달
		if (CheckIfReachedDestination())
		{
			_owner.OnReachDestination();
			return MoveResult.ReachedFinalDestination;
		}
		// 기다려야 하는 경우
		else if (_owner.CurrentPathNodes[_owner.CurrentPathIndex].waitTime > 0)
		{
			_owner.StartCoroutine(_owner.WaitAtNode(_owner.CurrentPathNodes[_owner.CurrentPathIndex].waitTime));
			_isWaiting = true;
			return MoveResult.Waiting;
		}
		// 노드 업데이트
		else
		{
			_owner.UpdateNextNode();
			return MoveResult.ReachedNode;
		}
	}
	return MoveResult.Moving;
}

// MoveResult.cs
public enum MoveResult
{
    None,
    Moving,
    Waiting,
    ReachedNode,
    ReachedFinalDestination,
    NoPath
}
```
