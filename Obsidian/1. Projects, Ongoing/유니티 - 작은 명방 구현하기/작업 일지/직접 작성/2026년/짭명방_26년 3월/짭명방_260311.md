## 버그 / 시스템 수정

### Operator가 적 판정하는 시스템
- 어제 "현재 타겟"의 유효성을 검증하는 메서드를 작업하다가 중단했음
- `bool`로 바꾸려고 했는데 어딘가 애매한 지점이 생겨서..

#### 구조 자체는 이렇게 변경
```cs
// CurrentTarget = null이 되어야 할 조건이라면 null로 변경
ValidateCurrentTarget(); 

if (CurrentTarget == null)
{
	SetCurrentTarget(); 
}
```

- 여기서 `ValidateCurrentTarget`은 처음엔 아래처럼 구성했는데
```cs
protected override void ValidateCurrentTarget()
{
	// null일 경우 처리
	if (CurrentTarget == null) return;
	
	// 비활성화됐을 때의 처리
	if (!CurrentTarget.gameObject.activeInHierarchy) 
	{
		CurrentTarget = null;
		return;
	}

	// 공격 범위에서 벗어난 경우
	if (!IsCurrentTargetInRange())
	{
		CurrentTarget = null;
		return;
	}
}
```
초기화해주는 조건마다 일일이  `CurrentTarget = null` 및 `return;`을 작성하는 게 별로라는 생각이 들었음. 늘어지기도 하고, 왜 `null`로 바꿔야 하지? 가 읽히지 않을 수도 있겠다는 생각(주석을 달면 되겠지만 여전히 늘어지는 게 문제)

- **이전에 고민했던 `bool` 메서드를 이 안에서 활용할 수 있다.**
```cs
protected override void ValidateCurrentTarget()
{
	// null일 경우 처리
	if (CurrentTarget == null) return;

	if (!IsValidTarget())
		CurrentTarget = null;
}

// CurrentTarget = null이 될 조건을 이 메서드 내부에 다 집어넣어줌
private bool IsValidTarget()
{
	// 비활성화됐을 때의 처리
	if (!CurrentTarget.gameObject.activeInHierarchy) return false;

	// 공격 범위에서 벗어난 경우
	if (!IsCurrentTargetInRange()) return false;

	return true;
}
```
> - 즉 `ValidateCurrentTarget()`은 `null`로 바꿀지 말지 결정만 하면 되고 `IsValidTarget`은 "왜 유효하지 않은지"에 대한 판단을 하므로 각 메서드가 한 가지 역할만 해야 하는 원칙에도 잘 부합한다. 이해하기도 쉽고.

### Medic - HP가 가득찬 팀원에게 힐이 나가는 현상
- 일단 `OpHealController.SetCurrentTarget`의 경우
```cs
CurrentTarget = _targetsInRange
	.Where(target => target.Health.CurrentHealth < target.Health.MaxHealth) 
	.OrderBy(target => target.Health.CurrentHealth / target.Health.MaxHealth)
	.FirstOrDefault();
```
> 원래 `Where`문이 없었는데 추가해줌

- 이외에도 체력 비율이 낮은 순서대로 힐을 주기 위해 `OrderBy`문도 수정했다.

> [!note]
> - 여기서 다시 위에서 해결했던 이슈에 대해 생각해봄
> - **`CurrentTarget == null`일 때만 `SetCurrentTarget`이 동작하면 안됨** - **`SetCurrentTarget`은 항상 동작해야 함**
> - 저 조건이 들어가는 건 현재 공격하고 있는 적이 죽어야만 공격 대상이 바뀌어야 한다는 말인데, 꼭 그렇지 않기 때문이다
> 	- 공격의 경우 목적지까지 남은 거리가 가장 짧은 적부터 우선 공격함
> 	- 힐의 경우 체력 비율이 가장 낮은 아군부터 우선 힐함
> - 예전에도 고민한 적 있는 내용인데 또 한 거 보면 **주석의 중요성을 새삼 느낀다**

- 그래서 `OpActionController`에서의 `CurrentTarget == null` 조건은 제외함
```cs
// 현재 타겟의 유효성 검사. 유효하지 않을 경우 CurrentTarget = null로 전환
ValidateCurrentTarget(); 

// SetCurrentTarget은 "항상" 실행됨
// 행동 대상을 체크하는 로직은 항상 동작해야 하기 때문임
SetCurrentTarget(); 
```

### MeteorSkill
- 얘는 작동되는 방식이 조금 다르다. 위는 `Controller`를 띄우고 그 안에서 구현되는 방식이고, 얘는 `OnActivated`에 로직이 다 들어가 있고 `Meteor` 파티클 자체에 컨트롤러가 들어가 있는 방식
```cs
// 중복 타격 방지를 위한 ID 세트
var enemyIdSet = new HashSet<int>();

// 범위 내의 모든 적에게 메테오 소환
foreach (Vector2Int pos in caster.GetCurrentSkillRange())
{
	Tile? tile = MapManager.Instance!.GetTile(pos.x, pos.y);
	if (tile != null)
	{
		foreach (Enemy enemy in tile.EnemiesOnTile.ToList())
		{
			if (enemyIdSet.Add(enemy.GetInstanceID()))
			{
				// 코루틴은 Monobehaviour을 받는 객체에서만 실행 가능
				// 이 스크립트는 ScriptableObject의 상속임. 실행 가능한 객체에게 요청한다.
				caster.StartCoroutine(CreateMeteorSequence(caster, enemy));
			}
		}
	}
}
```

- 그래도 적용하는 방식 자체는 동일함
1. 공격 대상 수집
2. 그 중 어떤 적을 선정할 것인가

- 다만 하나 체크할 점은 기존엔 고유 인스턴스 ID로 점검했다는 거 -> 그냥 `HashSet<Enemy>`를 써도 될 것 같음

#### 추가로..
- 숫자를 줄이는 것도 그렇지만, 아래 케이스도 생각해볼 필요는 있겠다.
1. `affectedTargets`의 갯수가 최대 6개를 초과하면 안 됨
2. 이전처럼 남은 거리를 기준으로 하는 로직을 쓸 경우, 번개가 내려치는 대상이 계속 뒤바뀌게 됨. 일반적으로 몹 하나를 빨리 잡는 방식이 더 효율적임
3. **최초 적용 대상을 계속 때리고, 이 적용 대상이 죽거나 범위를 빠져나갔을 때에만 다른 타겟을 찾도록 변경**

```cs
protected override void ApplyInitialEffect(UnitEntity target)
{
	if (target is Enemy enemy && caster != null)
	{
		// 슬로우 지속 시간은 장판이 사라지거나 벗어날 때까지이므로 무한으로 처리
		SlowDebuff slow = new SlowDebuff(float.PositiveInfinity, movementModifier);

		// 빈 자리가 있을 때에만 적용됨
		if (affectedTargets.Count < maxTargets)
		{
			enemy.AddBuff(slow);
			affectedTargets[enemy] = new List<Buff> { slow };
		}
	}
}
```
1. 인풋인 `target`은 범위 내에 있는 남은 거리가 가장 짧은 순서로 들어옴(위에 코드 있음)
2. `ApplyInitialEffect`에서는 영향을 받고 있는 대상 딕셔너리를 관리. 이 카운트가 `maxTargets`을 초과할 때만 새로운 `target`을 추가함
3. 나가는 경우는 기존에 이미 구현되어 있으므로(영역 내에 있는지 체크하고 없으면 슬로우 제거 & 딕셔너리에서 제거) 별도로 구현 X

- 이 정도면 될 것 같은데 싶어서 테스트 -> 잘 되는 듯
- `EnemyEasyWeak` 기준, 11마리의 몹이 나타났을 때 기존에는 스킬 하나로 완막 가능, 지금은 3마리 놓쳐서 실패함
