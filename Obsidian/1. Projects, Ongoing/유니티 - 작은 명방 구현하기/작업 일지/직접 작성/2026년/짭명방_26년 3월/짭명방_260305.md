## 이슈 수정

### 경로 막혔을 때 바리케이드까지의 경로를 계산하지 못하는 현상
- 점검 후 예상
	- 어제 수정한 내역 중, `Tile.IsWalkable`을 기준으로 체크한 부분이 있었음
	- 근데 목적지가 "바리케이드가 있는 타일"일 때는 `Tile.IsWalkable`이 `false`일 거임. 바리케이드가 그 내용을 바꿔버리니까

- 그러면 어떻게 수정해야 할까? 현재 상황은

1. 바리케이드가 최종 목적지일 때 : 경로에 포함시켜서 접근할 수 있어야 함
2. 최종 목적지가 아닐 때 : 경로에 포함되지 않아야 함

일단 현재의 `PathFindingManager.GetNeighbors`의 로직 중 일부
```cs
if (neighbor != null && neighbor.IsWalkable)
{
	// 대각선 이동
	if (x != 0 && y != 0)
	{
		// 대각선 이동 시, 양쪽 타일 모두 걸을 수 있어야 함
		Tile? SideA = currentMap.GetTile(tile.GridPosition.x + x, tile.GridPosition.y);
		Tile? SideB = currentMap.GetTile(tile.GridPosition.x, tile.GridPosition.y + y);

		if (SideA != null && SideB != null && SideA.IsWalkable && SideB.IsWalkable)
		{
			neighbors.Add(neighbor);
		}
	}
	// 직선 이동
	else
	{
		neighbors.Add(neighbor);
	}
}
```
> 현재 타일의 주변 8타일을 검사, 갈 수 있는 타일만 `neighbors`에 추가해서 반환하는 방식이다.
> 여기서 `neighbor.IsWalkable` 부분이 이슈인 거

>[!note]
>- 당장 생각나는 방법은 `GetNeighbors`에서 얻어지는 타일들은 `IsWalkable` 조건을 체크하지 않는 것 
>	- 즉 **최초에 도달할 수 있는 타일이라고 설정했으면 모두 `neighbors`에 추가**
>	- 하지만 해당 타일이 마지막 타일인지 여부는 저 메서드 내부에선 알 방법이 없기 때문에 외부에서 별도의 거르는 조건을 만드는 거임

- 사실 기존에 상위 메서드에선 이렇게 구현되어 있었음
```cs

// ...

foreach (Tile neighbor in GetNeighbors(currentTile))
{
	if (closedSet.Contains(neighbor)) continue; // 이미 포함된 타일
	
	// 여기서 점검함
	if (neighbor.HasBarricade() && neighbor != endTile) continue; // 마지막 타일이 아니면서 바리케이드가 있는 타일

	// ...
}
```

근데 어제 `GetNeighbors(currentTile)`에서 도달할 수 없으면 아예 걸러버렸기 때문에 발생한 현상으로 보임

- 테스트
	- ㅋㅋㅋㅋㅋㅋㅋㅋㅋ 아예 공중부양해서 센터로 지나감

#### 2차 수정
- 현재 상태에 관계 없이 최초에 해당 타일이 `IsWalkable`이었는지 여부를 점검해야 함
- 테스트에서 센터로 지나간 이유는 `GetNeighbors`에서 지나갈 수 있는 타일로 체크했고, 상위 메서드에서도 딱히 거르는 게 없었기 때문
- 나중에 코드를 다시 봤을 때 애매한 지점이 생길 수 있어서, 명확하게 주석으로 설명을 달아둠

```cs
// GetNeighbors

	// 최초에 걸을 수 있는 타일이라면 모두 포함함(바리케이드에 의해 현재 막힌 타일이더라도, 여기선 일단 포함하고 밖에서 거름)
	if (neighbor != null && neighbor.TileData.IsWalkable)
```

- 재테스트
	- 경로 관련 이슈는 해결, 다른 버그가 발생함

### Enemy가 바리케이드 공격할 때 NullReferenceException
- `Barricade.Despawn` 까지는 동작함

- 원인 발견
```cs
// DeployableUnitEntity.cs

public void Despawn(DeployableDespawnReason reason)
{
	// ...

	Undeploy(); // CurrentTile 해제 로직 포함
	HandleBeforeDisabled(); // 여기서 CurrentTile에 접근, 참조 없음 예외 발생
	
	// ...
}
```
- 서순 이슈라 두 메서드의 위치를 바꿔줌
