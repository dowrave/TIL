### 디버프 효과와 VFX 구분하기

- 반면 글로벌하게 쓸 수 있는 이런 류의 디버프 VFX는 어떻게 관리되어야 할까?

- 다만 **`Buff` 내부에서 `VFX`에 대한 참조를 가질 필요가 없다**는 것에 더 초점을 맞추면 됨
	- ~~원샷으로 쏘고 잊으면 되기 때문임~~
	- 더 정확히는 `BuffVFXManager`에서 추적하면 됨.
- 그리고 한 대상에 같은 종류의 VFX가 2개 이상 실행되는 건 분명히 이상하니까, `BuffVFXManager`에서 현재 재생 중인 VFX를 추적하도록 해야 함

#### 기능 분리 및 자세한 구현
1. 일단 `Debuff` 관련, 즉 `Stun`과 `Slow`은 명백한 `Debuff`이기 때문에 이름들을 싹 바꿨다. 

2. `Debuff`의 `enum` 타입을 정의한다.
```cs
public enum DebuffType
{
    None,
    Stun,
    Slow,
}
```

3. 일단은 `Buff` 내부애 묶을 것이므로 `Buff` 내부에 필드를 구현함
- 참고) 필드는 `virtual`로 구현할 수 없음 
	- `virtual`은 메서드, 프로퍼티, 인덱서, 이벤트에만 사용 가능.
- 대신 프로퍼티로 구현해주면 됨
```cs
// Base 클래스
public virtual DebuffType DebuffType => DebuffVFXType.None;

// 자식 클래스에서 override
public override DebuffType DebuffType => DebuffVFXType.Stun;
```

4. 버프가 제거될 때는 `DebuffVFXManager`에서는 상태 처리만 해둔다. `controller` 내부에서 자체적으로 스스로 반환되기 때문이다. 
```cs
// 이벤트 구독 해제
if (_activeVFX.TryGetValue(key, out var entry))
{
	// 이벤트 구독 해제
	SelfReturnVFXController vfxController = entry.obj.GetComponent<SelfReturnVFXController>();
	if (vfxController != null && entry.onExpired != null)
	{
		vfxController.OnDisabled -= entry.onExpired;
	}
	
	entry.obj.transform.SetParent(null); // 부모-자식 관계 해제
	
	// 기존에는 여기에도 ReturnToPool이 있었으나, 컨트롤러가 자체적으로 태그를 갖고 스스로 반환되기 때문에 그 부분은 제거함. 문제가 되면 추가.
	
	_activeVFX.Remove(key); // 딕셔너리 제거
}
```


### SelfReturnVFXController의 Initialize() 관련
```cs
    public void Initialize(float duration, Buff? sourceBuff = null, UnitEntity? caster = null)
```
> 기존 구현 방식

- 이게 구현할 때는 간편한데, 실사용할 때 헷갈린다. 어떤 건 null을 넣고, 어떤 건 넣지 않고 이런 일일이 구분하는 지점이 헷갈림. 
	- 코드를 항상 기억할 수 없기 때문에, **완전히 잊은 상태에서 다시 봤을 때에도 잘 읽혀야 한다.**
- 하지만 `duration`은 항상 필요한 상황에서, `Buff?` 와 `UnitEntity?`를 넣는 상황을 일일이 구분한다면 `Initialize()`만 4개를 구현해야 함

- `Claude`에게 물어봤는데, **이런 경우는 `Context 구조체`를 쓰는 방법이 있다고 한다.**

#### Context 구조체
- `Initialize()`을 4개를 만드는 대신, 1개로 통합한다.
- `Context` 구조체에서는 파라미터를 어떻게 받는지에 대한 정의를 해둔다.
```cs
public struct VFXInitContext 
{
	// 1. Initialize에 사용되는 필드들 명시
    public float Duration;
    public Buff SourceBuff;
    public UnitEntity caster;
    
    // 2. 조합에 따라 정적 팩토리로 제공함
	public static VFXInitContext FromBuff(Buff buff, float duration) => new ()
    {
        Duration = duration,
        SourceBuff =  buff
    };

    public static VFXInitContext FromCaster(UnitEntity caster, float duration) => new ()
    {
        Duration = duration,
        Caster = caster
    };

    public static VFXInitContext Full(UnitEntity caster, Buff buff, float duration) => new()
    {
        Duration = duration,
        SourceBuff = buff,
        Caster = caster
    };
}
```

- 이제 `Initialize()`는 아래처럼 구성한다.
```cs
// Initialize(float duration)에 가장 기본적인 동작을 정의했다고 가정

public void Initialize(VFXInitContext context) 
{
	if (context.SourceBuff != null) 
	{
		_sourceBuff = context.SourceBuff;
		_sourceBuff.OnRemovedCallback += ReturnToPool;
	}
	
	if (context.Caster != null) 
	{
		caster = context.Caster;
		caster.OnDeathStarted += ReturnToPool;
	}
	
	Initialize(context.Duration);
}
```

- 호출부에서는 아래처럼 불러옴.
```cs
vfx.Initialize(VFXInitContext.FromBuff(buff, 3f));
vfx.Initialize(VFXInitContext.Full(caster, buff, 3f));
```

> 비슷한 방식으로 `Builder` 패턴이 있다. 메서드 체이닝으로 객체를 생성하는 과정(Keras의 함수형 프로그래밍을 생각하면 쉬울 듯)인데, 지금 상황에선 오버엔지니어링이 될 수 있다고 함
