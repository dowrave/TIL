## 기타 이슈 수정

### ShieldVFX 위치 + VFX의 구현 관련
- 현재 : `ShieldSkill.OnActivated`에 위치.
```cs
public override void OnActivated(Operator op)
{
	// ...
	
	// Shield VFX 실행
	GameObject shieldVFXObject = ObjectPoolManager.Instance.SpawnFromPool(ShieldVFXTag, op.transform.position, Quaternion.identity);
	shieldVFXObject.transform.SetParent(op.gameObject.transform);
	ShieldVFXController shieldVFX = shieldVFXObject.GetComponent<ShieldVFXController>();
	if (shieldVFX != null)
	{
		shieldVFX.Initialize(duration, op);
	}
	
	// ...
```

- 왜 이슈인가? : `durationVFX`는 `OpSkillController.cs`에 구현되어 있음.
- 비슷하게 스킬이 지속되고 있음을 나타내는 VFX인데, 어떤 건 컨트롤러에, 어떤 건 `SO`에 구현되어 있기 때문에 일관성 이슈가 있음

필드를 추적하면서 관리하고 싶다면 `OpSkillController.cs`에 구현하는 게 맞겠는데, 구현하다보니 이런 생각도 들었다 : **굳이 추적해야 하나?** `Fire and Forget`이라는 게 있었는데, VFX의 수명은 오브젝트 자신이 관리하고, 그걸 생성한 주체는 만들기만 하고 잊는 개념을 적용할 수 있지 않을까?

- 마침 VFX 관련 요소들은 `SelfReturnVFXController.cs` 스크립트를 붙여놨다. 이걸 활용해봐야겠음.

>[!note]
>1. VFX를 생성하는 위치는 스킬 `SO`가 된다.  
>	- **`OnActivated`에 구현**해두면 될 듯.
>2. `OpSkillController.cs`에서는 `OnActivated`만 실행시키고, 내부 내용은 모른다.
>	- `VFX` 관련 로직들은 모두 제거함
>3. `Buff`가 생길 때 `VFX`도 함께 생기면 되고, `Buff`가 해제될 때 `VFX`도 함께 해제되면 된다. `Skill SO`는 그 둘을 이어주는 역할만 하는 느낌으로 가면 됨.
>	- 결국 버프 기반으로 VFX를 연결하는 느낌이 됐다. 근데 이건 구현하기 나름임

- 구조는 고쳤지만 `ShieldVFX`는 여전히 안 나타남
	- 생성되자마자 해제되는 듯?
	- 쉴드량 이슈는 아님 
	- 버프 생성될 때 `duration` 값이 들어오지 않기 떄문인 듯
- 착각하고 코드를 짰다.
	- `Skill`에서 어떤 버프가 적용될지에 대한 명시가 다 들어감
	- **`duration` 같은 필드는 `Buff`의 생성자에서 설정함.** 
		-  `OnApply`에서는 별도의 설정 없이 바로 적용하면 됨. 그래서 `OnApply`에서 `duration`을 받을 필요는 없다.
	- 이런 부분은 헷갈리기 쉬우니까 주석을 잘 달아놓자. 

- `Buff`에서 `duration`을 가져야 할까? 에 대한 생각.
	- `OpSkillController`에 의해 관리되는 `Buff`들은 사실 `duration`이라는 필드가 별도로 필요하진 않다. 스킬 자체가 `duration`을 지니고, 코루틴이 끝나면 버프도 함께 해제되기 때문임
	- **하지만 `Stun, Slow` 등의 즉발성 효과로 인한 디버프라면 얘기가 달라진다.** 얘네는 스킬과 연동되지 않고 효과가 들어간 시점부터 시간을 카운트하는 개념이기 때문이다. `BuffController`에서 관리되는 요소인 만큼, Buff 자체에서 `duration` 필드를 가지는 게 더 좋아 보임.

> 여러 곳에서 관리되더라도 `Skill -> Buff`의 흐름, `Skill`이나 `Buff`의 필드를 컨트롤러가 읽을 수 있는 구조이기만 하면 크게 상관은 없는 듯.
